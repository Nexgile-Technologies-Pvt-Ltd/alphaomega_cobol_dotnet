# 14. Known defects, ambiguities and open decisions

## Decision policy

This register separates source-observed behavior from proposed .NET behavior. It does **not** claim that any safe-target change has been implemented or approved.

| Term | Meaning |
|---|---|
| <code>DEF-*</code> | executable or declarative source has a demonstrable defect, unsafe edge, inconsistent state transition or malformed artifact |
| <code>DEC-*</code> | source is ambiguous, contradictory, environment-dependent, incomplete, or requires a product/operations choice |
| Strict parity | characterize the observed result and preserve it only where an explicit compatibility test/profile requires it |
| Safe default | proposed production behavior; it is not a claim about the COBOL and is not implemented merely because it is documented |

Statuses are deliberately conservative:

| Status | Meaning |
|---|---|
| **Proposed** | evidence and a safe disposition are documented; owner approval and implementation evidence remain outside this register |
| **Open** | business, data, security or operations ownership must choose a disposition |
| **Characterize** | supplied source is insufficient to determine a runtime result; test the deployed/compiler behavior or approve the stated safe rule |
| **Exclude candidate** | artifact appears demonstrative, stale or non-product; exclusion still requires scope approval |

Security weaknesses, data loss, unsafe partial commits, stale-memory reuse and unbounded/recursive control flow are never production compatibility switches. A strict harness may inject or simulate their ordering without persisting unsafe state. Claim labels and source precedence are defined in [Documentation conventions](Documentation-Conventions.md#claim-labels).

## Atomicity policy

The legacy order is part of characterization, but partial commits are not the production default. Core CSD files are nonrecoverable/nonjournaled ([CARDDEMO.CSD lines 1–99](../Old_Cobol_Code/app/csd/CARDDEMO.CSD#L1-L99)); a source <code>SYNCPOINT ROLLBACK</code> therefore does not prove every VSAM mutation can be undone.

| Use case | Observed mutation order/risk | Strict parity | Safe-target boundary | Register IDs |
|---|---|---|---|---|
| account/customer update | account rewrite, then customer rewrite; rollback attempted only after customer failure ([COACTUPC lines 3888–4193](../Old_Cobol_Code/app/cbl/COACTUPC.cbl#L3888-L4193)) | fault-inject between writes | account + customer together, optimistic version checks | DEF-ONL-001, DEF-ONL-005 |
| card update | read/update and rewrite one card, but relationship and CVV state can be wrong | characterize validation/write order | verified relationship + complete card mutation | DEF-ONL-003/004 |
| transaction add | one transaction write after nonserialized highest-ID scan | concurrent collision test | validation + ID allocation + insert | DEF-ONL-007/009 |
| bill payment | transaction write, balance subtraction, account rewrite; errors can continue | boundary fault tests | payment transaction + account balance together | DEF-ONL-008/009 |
| report request | many TDQ line writes ending in marker | exact strict JCL formatter | one durable request/outbox record | DEF-ONL-010/011 |
| posting | category balance, account, transaction; no rollback | exact per-record ordering | one transaction per input row plus run ledger | DEF-BAT-001/002/010 |
| interest | transactions written per category; account applied on account change; final account skipped | exact fixture oracle | one account/cycle transaction with idempotent charge keys | DEF-BAT-005/007/010 |
| combine/output generation | new generations can collide or publish partially | file-order characterization | temp output, close/fsync, atomic publish | DEF-BAT-010 |
| branch export/import | no cross-file snapshot; multiple destructive outputs | byte fixture and failure injection | consistent snapshot; validate all destinations; atomic import/publication | DEF-XFR-003/007 |
| transaction type/category | row mutation syncpoints, later independent export/load | characterize DB2/VSAM lag | relational mutation transaction; consistent paired export | DEF-OPT-TT-007, DEC-OPT-TT-001 |
| authorization decision | NO_SYNCPOINT get/reply, reply before IMS root/detail | ordering only; never persist data-loss mode | inbox + summary + detail + outbox transaction, idempotent ack/delivery | DEF-OPT-AUTH-001/002 |
| fraud toggle | linked Db2 operation then IMS replace and CICS syncpoint ([COPAUS1C lines 520–569](../Old_Cobol_Code/app/app-authorization-ims-db2-mq/cbl/COPAUS1C.cbl#L520-L569)) | preserve coordinated outcome | fraud history + detail in one relational transaction | DEC-OPT-AUTH-003 |
| account/date inquiry | MQGET/MQPUT use CICS syncpoint but fixed destinations/IDs | descriptor/byte profile | validated request + reply/outbox; bounded worker | DEF-OPT-MQ-001/002/003 |

Safe-mode restart tests terminate at every persistence, acknowledgement and publication boundary. Success means no duplicate financial effect, no lost accepted request, no partially published aggregate, and a defined retry/dead-letter outcome. This is a proposed policy, not an implementation claim.

## Account update layout shift

**DEF-ONL-001** is byte-deterministic. The canonical 300-byte account record is ([CVACT01Y lines 4–17](../Old_Cobol_Code/app/cpy/CVACT01Y.cpy#L4-L17)):

| One-based bytes | Canonical field |
|---:|---|
| 1–102 | identity/status, balances, limits, three dates and two cycle amounts |
| 103–112 | <code>ACCT-ADDR-ZIP</code> |
| 113–122 | <code>ACCT-GROUP-ID</code> |
| 123–300 | filler |

The custom <code>ACCT-UPDATE-RECORD</code> omits ZIP, places group immediately after byte 102, and expands filler from 178 to 188 ([COACTUPC lines 418–433](../Old_Cobol_Code/app/cbl/COACTUPC.cbl#L418-L433)). A rewrite using that group therefore puts group in canonical ZIP bytes 103–112, while canonical group bytes 113–122 are part of filler. This is not a speculative offset: it follows directly from the two PIC layouts.

- **Strict parity:** a dedicated codec/fault fixture may demonstrate the shifted bytes; no production compatibility option writes them.
- **Safe default:** deserialize and update by named canonical fields, preserve ZIP unless explicitly changed, and write group at bytes 113–122 only for legacy export.
- **Decision:** migration owner must determine whether previously updated source records require an audited ZIP/group repair. Status **Open**; no repair is claimed.

## Interest final-account update

**DEF-BAT-005** results from loop control, not rounding. <code>CBACT04C</code> applies the previous account only when a later account key is encountered. The final read sets EOF inside an iteration that entered with EOF false, so the outer ELSE intended to update at EOF is not reached before the loop exits ([CBACT04C lines 188–221](../Old_Cobol_Code/app/cbl/CBACT04C.cbl#L188-L221)). Interest transaction writes have already occurred in [lines 462–515](../Old_Cobol_Code/app/cbl/CBACT04C.cbl#L462-L515).

For the supplied deterministic fixture and cycle <code>2022071800</code>:

- 50 interest transactions total 1,279.16;
- only 1,260.39 is applied to accounts;
- final account 50's 18.77 transaction exists, but its balance is not increased and its cycles are not reset;
- accounts 1–49 are applied/reset.

The oracle is derived on [Batch Processing](05-Batch-Processing.md#strict-interest-result-after-posting).

- **Strict parity:** retain a named <code>Interest.SkipFinalAccountRewrite</code> characterization profile/golden result only.
- **Safe default:** group deterministically and commit each account's charges plus balance/reset exactly once, including the final group.
- **Decision/status:** proposed safe disposition; implementation and financial-owner approval are not asserted.

## Export key offset

**DEF-XFR-004.** <code>EXPORT-SEQUENCE-NUM</code> begins at one-based byte 28, so the four bytes occupy 28–31 (zero-based offset 27) in [CVEXPORT](../Old_Cobol_Code/app/cpy/CVEXPORT.cpy#L1-L103). [CBEXPORT.jcl lines 30–33](../Old_Cobol_Code/app/jcl/CBEXPORT.jcl#L30-L33) defines <code>KEYS(4 28)</code>, one byte later.

- **Strict parity:** the supplied 500-record sequential fixture proves header offsets and begins with binary sequence 1; malformed-VSAM behavior requires environment characterization if it remains in scope.
- **Safe default:** identity/order uses declared bytes 28–31; import validates monotonic/full sequence separately from storage key.
- **Status:** Proposed; any demand to reproduce the malformed VSAM key is Open.

## Release decision gate

Before production cutover, every **Open** or **Characterize** item must have an owner, evidence/decision date, approved disposition and linked implementation/test evidence. A status may move to closed only in a future revision that cites those artifacts. This snapshot intentionally contains no closed claims.

Minimum gates:

1. financial/accounting owners approve posting debit arithmetic, interest rounding/final-account correction, authorization available-credit initialization and added validations;
2. security owners approve credential migration, authorization matrix, secret handling, audit and retention;
3. operations owners approve canonical batch graph, restart/idempotency, provider/HA, backup/recovery, queues, external adapters and retention;
4. migration owners approve layout/data repairs for account ZIP/group, xref filler, card/xref relationships, branch export key and IMS unloads;
5. characterization tests capture every retained strict quirk, while safe-mode fault tests prove atomicity and bounded failure behavior.
