# Documentation conventions

[← Home](Home.md) · [Product scope →](01-Product-Scope.md)

## Goal

These conventions prevent proposed .NET design, repository prose, and analyst interpretation from being mistaken for behavior proven by the COBOL estate.

## Claim labels

| Label | Meaning | May define parity behavior? |
|---|---|---:|
| **Observed — code** | Directly implemented by COBOL, assembler, BMS, DDL/DBD/PSB, JCL, CSD, procedure, scheduler, or script | Yes |
| **Observed — data** | Directly present in supplied sample data | Only as a fixture/example, not as a universal rule |
| **Observed — repository prose** | Stated in a shipped README or diagram | Only when executable/declarative artifacts do not contradict it |
| **Derived** | Deterministic conclusion from cited observed evidence | Yes, with the derivation stated |
| **Target recommendation** | Proposed behavior or structure for the .NET implementation | No; it becomes normative only after project approval |
| **Decision required** | Contradiction, missing dependency, ambiguous semantics, hard-coded environment value, placeholder, or incomplete behavior | No |

## Source precedence

When artifacts disagree, this wiki uses the following precedence for the parity baseline:

1. Executable COBOL/assembler behavior and embedded SQL/DLI/MQ calls.
2. Declarative runtime definitions: BMS, CSD, DDL/DCL, DBD/PSB, JCL and procedures.
3. Supplied data layouts and sample records.
4. Scheduler and developer automation.
5. Repository README files and diagrams.

This precedence does not hide discrepancies. Every material conflict belongs in [Known Defects, Ambiguities and Open Decisions](14-Known-Defects-and-Open-Decisions.md#source-contradictions).

## Deep-link format

Source evidence is linked to the file and, for text sources, to the relevant line or line range. Example:

`[sign-on role routing](../Old_Cobol_Code/app/cbl/COSGN00C.cbl#L221-L238)`

Each requirement has a stable identifier. Prefixes are:

| Prefix | Scope |
|---|---|
| `FR-AUTH` | authentication/session behavior |
| `FR-ACCT` | account behavior |
| `FR-CARD` | card behavior |
| `FR-TRAN` | transaction behavior |
| `FR-RPT` | reports/statements |
| `FR-BILL` | bill payment |
| `FR-USER` | security-user administration |
| `FR-BATCH` | batch behavior |
| `FR-OPT` | optional modules |
| `NFR` | non-functional compatibility requirements |
| `DATA` | data contract requirements |
| `OPS` | operational requirements |

Anchors use lower-case kebab-case and remain stable even if headings are reworded.

## Exact-parity versus safe-target behavior

The source contains legacy quirks, security weaknesses, hard-coded values, and incomplete code. The wiki therefore records two tracks:

- **Exact-parity track:** behavior needed by a characterization test or a strict migration.
- **Safe-target track:** recommended production behavior for the .NET product.

The safe-target track must not silently change a parity rule. Each deliberate deviation needs a decision record and a test for the chosen behavior.

## Numeric and date interpretation

COBOL `PIC` clauses are recorded verbatim. .NET mappings use `decimal`, integer, string, or date/time types only on target-design pages. A source field declared as text remains text in the parity contract even when its contents look like a date. Signed display numeric sample data may use EBCDIC/ASCII overpunch; parsing rules are documented per layout rather than inferred from visual appearance.
